# FoodMate-AI —— 基于多智能体协作的智能外卖平台

# 第一部分：设计思路

## 一、项目背景与问题分析

当前主流外卖平台在智能化服务方面存在三个显著不足。第一，推荐系统对场景因素的感知严重缺失。传统的协同过滤或基于内容的推荐算法仅依赖用户的历史行为数据，却忽略了天气、交通、时段、节气等一系列直接影响用户点餐决策的环境因素。例如，一个寒冷的雨天和一个晴朗的夏日，用户对餐食的需求显然截然不同，但传统系统给出的推荐结果往往没有差别。第二，菜品定价策略过于僵化。绝大多数餐饮商家仍然采用固定价格模式，无法根据供需变化、销量趋势等市场信号动态调整价格，既错过了畅销品的利润空间，也无法通过降价策略挽救滞销品。第三，用户在点餐过程中缺乏健康指导，面对一份菜单无法快速了解各菜品的热量、食材成分和过敏原信息，对有特殊饮食需求（如糖尿病、食物过敏）的用户尤其不友好。

基于以上分析，本项目提出 FoodMate-AI 智能外卖平台，以人工智能技术为核心驱动力，从推荐、定价和健康三个维度全面提升外卖服务的智能化水平。

## 二、整体架构设计理念

本系统的架构设计遵循"微服务 + AI 中台 + 事件驱动"的三层设计理念。在微服务层面，我们将整个平台拆分为九个独立的微服务，其中六个使用 Java Spring Boot 构建，负责用户认证、商家管理、订单处理、营销优惠、平台运营和用户画像等核心业务逻辑；另外三个使用 Python FastAPI 构建，分别承担智能推荐、AI 动态定价和多模态营养分析的 AI 能力。之所以采用 Java 和 Python 双语言技术栈，是因为 Java 生态在企业级微服务治理方面有着成熟的解决方案（Spring Cloud 全家桶提供了服务注册发现、集中配置管理、服务间调用等基础能力），而 Python 在 AI/ML 生态方面具有无可比拟的优势（LangGraph、LangChain、OpenAI SDK 等前沿框架均以 Python 为主）。这种异构架构让每个服务都能使用最适合其职责的技术栈。

在 AI 中台层面，三个 Python 服务并非简单的 API 封装，而是各自拥有完整的智能体架构。推荐服务内部编排了三个智能体（环境感知、用户画像、MAB 决策），形成一条多智能体协作流水线；定价服务构建了"数据采集 → AI 分析 → 提案审批"三阶段自动化流水线；营养分析服务则利用多模态视觉模型实现了从图片输入到结构化营养信息输出的端到端处理链路。

在事件驱动层面，系统通过 RabbitMQ 消息队列实现服务间的异步解耦。以订单支付流程为例：当用户完成支付后，订单服务通过 RabbitMQ 的 `order.events` exchange 发布 `order.paid` 事件，AI 定价服务作为消费者监听该事件并将销售数据写入自己的数据库，同时订单服务通过 HTTP 同步调用平台服务触发佣金计算。这种同步与异步结合的通信模式，既保证了关键业务流程（如佣金计算）的实时性，又避免了 AI 数据采集对主流程的阻塞。

## 三、数据架构设计

在数据存储层面，系统采用多数据库混合架构的设计策略。PostgreSQL 作为主数据库承载所有结构化业务数据，其中主库 `food_delivery_db` 包含用户表（users）、商家表（merchants）、菜单表（menu_items）、订单表（orders）、订单明细表（order_items）、优惠券模板表（coupon_templates）、用户优惠券表（user_coupons）、平台服务表（platform_services）、商家订阅表（merchant_service_subscriptions）、结算单表（merchant_settlements）、佣金记录表（commission_records）等超过二十张业务表。AI 定价服务使用独立的 `ai_pricing_db` 数据库存储销售历史（sales_history）和定价提案（pricing_proposals），实现了 AI 数据与业务数据的物理隔离，避免 AI 分析任务的大量读写操作影响主库性能。

MongoDB 则专门用于用户画像服务，存储用户的偏好、行为记录、浏览历史等非结构化文档数据。选用 MongoDB 的原因在于用户画像数据的 Schema 高度灵活——不同用户可能拥有完全不同维度的偏好标签和行为记录，文档数据库的灵活 Schema 天然适合这种场景。Redis 作为全局缓存层，为热点数据查询提供毫秒级的响应速度。

## 四、推荐系统的智能体设计思路

推荐系统是本项目最核心的创新点，其设计思路是将推荐过程建模为一个多智能体协作的决策问题，而非传统的单一算法打分排序。我们设计了三个专职智能体，每个智能体负责收集和分析一个维度的信息，最终由决策智能体综合所有信息做出排序决策。

环境感知智能体（ContextAgent）负责感知和分析当前的外部环境信息。它通过并行调用和风天气 API 和高德地图 API（使用 `asyncio.gather` 实现异步并发），同时获取天气状况、交通拥堵程度和时间信息。在天气分析方面，该智能体不仅获取温度、湿度、天气状况等基础数据，还会进一步计算配送延迟影响（雨天估算延迟 20-60 分钟）、温度驱动的食物推荐（低于 10°C 推荐火锅热汤，高于 30°C 推荐凉皮冰品）、以及基于温湿度的人体舒适度指数。在交通分析方面，智能体根据拥堵指数动态调整推荐的配送半径——拥堵指数 1.5 时推荐 5 公里范围内的餐厅，拥堵指数超过 2.5 时缩小到仅推荐 1 公里范围内的餐厅。最终，环境感知智能体将所有环境因素综合为一个环境影响评分，权重分配为天气 40%、交通 40%、高峰时段 20%。

用户画像智能体（ProfilerAgent）从 MongoDB 画像数据库中提取用户的历史偏好、饮食限制、口味倾向等信息，并进行意图分析和用户分群。意图分析会检测用户查询中的菜系关键词（如"川菜""日料"）、情绪关键词（如"想吃辣的""清淡点"）、紧急度（如"快"），以及饮食限制（如"不要花生""低糖"）。用户分群则将用户划分为预算型（budget）、标准型（standard）和高端型（premium）三个群组，不同群组的推荐偏好权重有显著差异：预算型用户的价格权重高达 0.35，而高端型用户的评分权重为 0.35。更重要的是，画像智能体会根据环境感知智能体的分析结果动态修正用户偏好——例如恶劣天气时自动缩短用户的距离偏好并增加热食推荐权重。

决策智能体（DecisionAgent）是整个推荐系统中代码量最大（1392 行）、算法最复杂的核心组件。它实现了四种多臂老虎机（Multi-Armed Bandit, MAB）策略。其中上置信界算法（UCB1）通过 exploitation + exploration 的方式平衡已知最优选择与未探索选择之间的权衡；汤普森采样（Thompson Sampling）使用贝叶斯 Beta 分布采样，对每个餐厅进行 10 次采样取平均值，天生具有自适应的探索-利用平衡；ε-Greedy 策略以 0.1 的概率随机探索新餐厅。而系统默认使用的上下文感知老虎机（Contextual Bandit）是最核心的策略，它将每个餐厅的推荐分数分解为基础分（0.50）、变量分数（距离 25% + 评分 30% + 价格匹配 20% + 菜系匹配 20% + 配送时间 10%）和上下文奖励三个部分。上下文奖励部分根据天气影响（±0.20）、温度影响（±0.18）、交通影响（±0.12）、运动后状态（±0.22）、以及用户意图匹配（+0.40，"一票否决级"权重）等因素动态加成，最终实现了一个真正能感知上下文的推荐排序算法。所有 MAB 策略均支持在线学习：用户的点击行为给予 0.3 倍奖励，下单行为给予 1.0 倍奖励，评分行为给予 0.5 倍奖励，使推荐系统能够根据用户的实时反馈持续自我优化。

三个智能体的协作编排采用 LangGraph 的 StateGraph 状态机实现。常规流程为"环境分析 → 画像分析 → 决策排序"的串行管线，但系统设计了条件路由机制——当检测到用户的请求带有紧急标记时，流程会跳过画像分析直接进入决策阶段，以最短路径返回推荐结果。此外，系统还实现了一套并行编排器（ParallelOrchestrator），支持环境感知、POI 检索和画像分析三个任务同时并行执行，最后由推理智能体（ReasoningAgent）合并各路结果并生成最终推荐，进一步降低了端到端的推荐延迟。

## 五、端云协同隐私保护设计

在推荐系统之上，本项目还设计了一套端云协同的隐私保护推荐架构。这一设计的核心思想是：用户的语音输入和健康数据（步数、心率等）属于高度敏感的隐私信息，不应直接上传到云端。因此，系统在移动端部署了两个端侧 AI 模型：一个是 Vosk 离线语音识别模型（vosk-model-small-cn），用于将用户的语音输入离线转换为文字，全程不经过网络；另一个是 Qwen-2.5-0.5B-Instruct 端侧大语言模型（Q4 量化、GGUF 格式，通过 llama.rn 框架运行在手机本地），用于将语音转换后的自然语言文本提纯为结构化的 JSON 约束（包含 `forbidden_ingredients`、`required_temperature`、`preferred_tags`、`max_price` 等字段）。经过端侧处理后，系统仅将脱敏的结构化约束发送到云端，云端的多智能体推荐系统基于这些约束完成推荐，实现了"敏感数据不出端、推荐能力不打折"的隐私保护目标。

端侧的语音处理管线还集成了智能静音检测机制——2 秒短停顿为语句间歇、5 秒长停顿自动结束录音，并通过三通道事件拦截（NativeEventEmitter + DeviceEventEmitter + Vosk 官方包装器）确保跨平台兼容性和事件捕获的可靠性。健康上下文方面，系统通过手机硬件步数传感器实时采集步数数据，如果检测到用户 30 分钟内步数超过 2000 步则判定为"刚运动完"状态，此状态会在 5 分钟后自动重置。运动后状态会作为上下文信息传递给推荐系统的决策智能体，后者在 Contextual Bandit 算法中为此给予 ±0.22 的上下文奖励加成，从而推荐更适合运动后食用的高蛋白、易消化餐食。

## 六、AI 动态定价设计思路

AI 动态定价系统的设计遵循"数据驱动 + AI 决策 + 人机协同"的理念。系统通过 RabbitMQ 实时监听订单支付事件来自动采集销售数据，无需商家手动录入。当定价分析任务被触发时（默认每周一次，也支持手动触发），系统会通过 HTTP 接口从商家服务拉取菜单数据、从订单服务拉取销售统计，然后将这些数据连同菜品的历史价格一起提交给 Google Gemini 大语言模型进行分析。Gemini 在 prompt 中被要求扮演"餐厅收益管理总监"角色，根据过去 7 天的销量和营收数据做出三类决策：滞销品（周销量低于 5 份）适用降价促销策略（MARKDOWN），畅销品（周销量超过 50 份）可适当涨价（SURGE），表现稳定的菜品维持现价（MAINTAIN）。AI 给出的建议并非直接生效，而是以"定价提案"的形式保存，商家可以配置自动审批（设定价格变动阈值，阈值内自动执行）或人工审批（商家在后台逐条确认或拒绝）。这种人机协同的模式既发挥了 AI 的数据分析能力，又保留了商家对自身经营的最终决策权。

## 七、营销系统的算法设计

营销系统的核心算法是优惠券组合优化，其设计思路源自经典的 0/1 背包问题的变种。系统支持四种优惠券类型：折扣券（DISCOUNT）、满减券（THRESHOLD_REDUCTION）、无门槛券（NO_THRESHOLD）和免运费券（FREE_SHIPPING），并且不同优惠券之间存在复杂的叠加与互斥关系。算法首先将用户持有的所有可用券分为"可叠加"和"互斥"两组，然后分别生成两类候选方案：第一类方案仅包含可叠加券的所有子集组合（通过位掩码枚举实现），计算每种组合的总优惠金额；第二类方案选择一张互斥券，再搭配与之兼容的可叠加券。最终取总优惠金额最大的方案作为推荐结果返回给用户。

在此基础上，系统还实现了智能发券引擎，通过规则引擎定义自动发券条件。触发类型包括新用户注册（NEW_USER）、信用等级升级（CREDIT_UPGRADE）、订单里程碑达成（ORDER_MILESTONE，如第 10 单、第 50 单）和 VIP 月度福利（VIP_USER）。每条规则可以配置触发频率（ONCE——仅触发一次、DAILY/WEEKLY/MONTHLY——周期性触发），并设有总发放上限和单用户上限，防止过度发放。

## 八、平台运营与结算设计

平台服务的设计围绕"增值服务订阅 + 订单分成 + 自动结算"三个核心流程。平台提供多种增值服务供商家订阅，包括基础技术服务费（按订单金额的 3% 收取）、配送服务（8%）、流量推广（首页推荐位，¥50/月固定费用）和运营工具（数据报表，¥20/月固定费用）。佣金计算支持三种计费方式：按比例（PERCENTAGE）、按单固定（FIXED_PER_ORDER）和包月固定（FIXED_MONTHLY）。

每笔订单完成后，平台服务会遍历该商家订阅的所有按单计费服务，逐一计算分成金额并生成佣金记录。当发生退款时，对应的佣金记录会被标记为 REFUNDED 并回滚。结算方面，系统支持按周和按月两种结算周期，自动生成包含 GMV、总分成、净收入等明细的结算单，结算单号格式为 `ST{年月}{W或M}{商家ID后3位}{流水号}`。商家收到结算单后有 3 天确认期，可以选择确认或提出异议；超过确认期限未操作的结算单会被系统自动确认。管理员可以对有争议的结算单进行金额调整，并执行批量打款操作。

# 第二部分：实施方案

## 一、技术选型与基础设施搭建

项目选用的技术栈覆盖了从底层基础设施到上层应用的完整链路。后端 Java 服务统一使用 Java 21 和 Spring Boot 3.2 版本，引入 Spring Cloud 2023.0 版本作为微服务治理框架，包括 Eureka 实现服务注册与发现、Spring Cloud Config 实现集中配置管理、OpenFeign 简化服务间 HTTP 调用。安全认证统一采用 JWT（使用 JJWT 0.11.5 库），全局配置中设定了统一的密钥和 24 小时过期时间。数据持久层使用 Spring Data JPA 配合 PostgreSQL 15，用户画像服务使用 Spring Data MongoDB 配合 MongoDB 6.0。

Python AI 服务统一使用 FastAPI 框架，其异步特性非常适合调用外部 AI API 时的 I/O 等待场景。推荐服务的依赖包包括 LangGraph（多智能体状态图编排）、LangChain（LLM 调用链）、OpenAI SDK（兼容 Gemini/DeepSeek API）、FastMCP（MCP 协议服务器）、Pandas 和 NumPy（数据处理）、aiohttp（异步 HTTP 客户端）。AI 定价服务使用 SQLAlchemy（异步 ORM，配合 AsyncPG 驱动）、aio-pika（异步 RabbitMQ 客户端）和 APScheduler（定时任务调度）。营养分析服务依赖 httpx（异步 HTTP 客户端）用于调用 Gemini Vision 多模态 API。

移动端采用 React Native 0.83 配合 React 19.2 和 TypeScript，页面导航使用 React Navigation 7，HTTP 通信使用 Axios。端侧 AI 方面集成了 llama.rn（端侧 LLM 推理框架，运行 Qwen-2.5-0.5B-Instruct 的 Q4 量化 GGUF 模型）和 react-native-vosk（离线语音识别）。此外还集成了 react-native-image-picker（NutriVision 拍照功能）和 react-native-geolocation-service（GPS 定位用于 POI 检索）。Web 管理端使用 React + Vite 构建，面向平台管理员和商家。

整个系统通过 Docker Compose 实现一键编排部署，compose 文件长达 484 行，定义了全部基础设施服务（PostgreSQL、MongoDB、RabbitMQ、Redis）、九个应用服务和三个可观测性服务（Prometheus、Grafana、Zipkin）。所有数据库初始化脚本放置在 `backend/scripts/` 目录下，通过 PostgreSQL 的 `docker-entrypoint-initdb.d` 机制在容器首次启动时自动执行，共有 15 个 SQL 脚本负责创建表结构和插入种子数据。

## 二、多智能体推荐系统的实施

推荐服务的入口文件 `main.py` 定义了 FastAPI 应用，应用启动时通过 lifespan 事件初始化 LangGraph 编排器和各个智能体实例。API 路由分为两组：`/agents` 前缀的多智能体推荐 API 和 `/api/v2/mcp` 前缀的 MCP 协议 API。

环境感知智能体的实现中，天气分析模块接入了和风天气 QWeather API，支持 JWT 认证和传统 API Key 两种认证方式，并实现了智能降级——当 API 不可用时会基于当前时间和季节生成模拟天气数据，保证推荐流程不会因外部 API 故障而完全中断。交通分析模块调用高德地图 API 获取路况信息，返回的拥堵指数直接影响推荐餐厅的距离筛选范围。时段分析模块根据当前小时数判定用餐时段类型（早餐、午餐、下午茶、晚餐、夜宵、非用餐时段），不同时段会影响推荐菜系的权重分配。

用户画像智能体在获取到 MongoDB 中的用户画像文档后，会构建一个完整的 `UserProfile` 对象，包含偏好菜系（preferred_cuisines）、价格区间（price_range）、口味偏好（taste_preferences）、饮食限制（dietary_restrictions）、用户分群标签（user_segment）、时间敏感度（time_sensitivity）、各菜系历史下单频率（cuisine_order_frequency）和近期浏览过的商家列表（browse_merchants）。画像智能体最重要的输出是一组动态推荐权重——这些权重不是固定值，而是根据用户分群和当前环境上下文实时计算的。例如预算型用户在恶劣天气条件下，距离权重会被进一步提高（因为希望快速送达），同时价格权重维持较高水平。

决策智能体的 Contextual Bandit 算法实现中，每个候选餐厅的最终推荐分数经过四层计算。第一层是固定基础分 0.50，确保每个餐厅都有一个底线分数。第二层是变量分数，基于餐厅自身属性（距离、评分、价格匹配度、菜系匹配度、预估送达时间）加权计算后再乘以 0.3 系数。第三层是上下文奖励，这是 Contextual Bandit 的核心——算法会根据当前环境上下文对每个餐厅给予差异化的加成或减成，包括天气是否适合该餐厅的菜系（±0.20）、温度是否匹配（±0.18）、交通状况是否有利于该距离的配送（±0.12）、用户是否处于运动后状态（±0.22，运动后偏好高蛋白食物）以及用户意图是否与餐厅强匹配（+0.40，这是权重最高的单因素，意味着如果用户明确表达"想吃川菜"，川菜餐厅会得到压倒性的加分）。第四层是历史奖励分，基于该餐厅过往的 MAB 平均奖励值乘以 0.05 的小系数，确保算法在保持对新信息敏感的同时也会参考历史表现。

推荐结果生成后，系统还会调用 DeepSeek 大语言模型（deepseek-chat，temperature=0.8，max_tokens=1200）为每个推荐餐厅生成个性化的推荐理由文案。当 DeepSeek API 不可用时，系统会降级为基于规则的快速理由生成，使用 emoji 标签（如 🏆 表示高评分、🌧️ 表示雨天适合、❄️ 表示低温暖食推荐）拼接推荐理由。

在端云协同推荐的 API 实现中（`POST /agents/edge-synergy-recommend`），系统接收前端发送的结构化约束 JSON，在 DecisionAgent 执行 MAB 排序之前先进行硬过滤——将包含用户禁忌食材（forbidden_ingredients）的餐厅直接排除，将不符合温度需求（required_temperature）的餐厅过滤掉，然后再对剩余餐厅执行上下文感知的 MAB 排序。如果过滤后无餐厅匹配，返回 `NO_MATCH` 标记，前端收到此标记后会自动降级为普通推荐。

推荐服务还实现了完整的 MCP（Model Context Protocol）协议层。增强型 MCP 服务器（enhanced_mcp_server.py，879 行）基于 FastMCP 框架发布了 8 个标准化工具（analyze_environment、get_weather_analysis、get_traffic_analysis、analyze_user_profile、get_user_preferences、get_smart_recommendations、make_mab_decision、update_recommendation_feedback 等）和 3 个 Agent 能力描述资源（agent://context-agent/capabilities、agent://profiler-agent/capabilities、agent://decision-agent/capabilities），支持 stdio 传输模式。另外还实现了 MCP over HTTP/SSE 服务器（mcp_http_server.py，943 行，端口 8003），增加了 `get_parallel_recommendations` 工具以支持并行推荐请求。

## 三、AI 智能定价系统的实施

AI 定价服务的事件处理模块通过 aio-pika 异步 RabbitMQ 客户端连接到消息队列，声明 `order.events` exchange 并绑定 `order.paid` routing key。每收到一条订单支付事件，消费者函数将事件中的菜品明细（菜品 ID、数量、单价、商家 ID）逐条写入 `sales_history` 表，为后续的 AI 分析积累数据。

定价分析周期由 APScheduler 定时触发，默认间隔为 604800 秒（即 7 天一次），也可通过 API 端点手动触发。分析流程如下：首先通过 HTTP 调用商家服务的内部接口获取所有开启了动态定价功能的活跃商家列表；然后对每个商家并行拉取其菜单数据和近 7 天的销售统计；接着将菜品名称、当前价格、7 天销量和营收数据组装成 prompt，提交给 Gemini AI（通过引力 AI 中转的 OpenAI 兼容接口调用）进行分析；Gemini 的分析结果以 JSON 格式返回，包含建议价格（suggested_price）、策略类型（strategy_type）和分析理由（reasoning）；系统将每条分析结果保存为一条 PricingProposal 记录，状态根据商家配置设为 PENDING（待人工审批）或 AUTO_APPROVED（自动通过）。

自动审批的逻辑是：如果商家在后台开启了 `enableAutoApproval` 开关，并且 AI 建议的价格变动幅度在商家设定的 `autoApprovalThreshold` 阈值范围内，提案将被自动批准执行。自动批准的提案通过 RabbitMQ 的 `pricing.events` exchange 发布到 `merchant.pricing.updates` 队列，商家服务的 `PricingEventConsumer` 监听该队列后，根据消息中的 routing key 判断处理方式：如果包含 "auto" 关键字则直接调用 MenuService 更新菜品真实价格并发送通知给商家；否则将提案存入 `price_change_proposals` 待审批表并通知商家前往后台审阅。商家可以在移动端或 Web 端的智能定价管理页面查看所有待审批提案，逐一批准或拒绝。

为了方便开发测试，数据库初始化脚本中预设了四种典型的销量趋势模拟数据：红烧牛肉面的销量从每日 20 份严重下滑到 2 份（触发 MARKDOWN 策略），酸菜肉丝面的销量保持在 8-12 份（触发 MAINTAIN 策略），秘制烤五花的销量从 5 份暴涨到 30 份（触发 SURGE 策略），超级至尊披萨的销量持续低迷在 0-3 份（触发 MARKDOWN 或 MAINTAIN 策略）。这些模拟数据使开发者无需积累真实订单就能验证整条 AI 定价流水线的正确性。

## 四、NutriVision 营养分析的实施

NutriVision 服务对外暴露唯一的 API 端点 `POST /api/v1/vision/analyze`，接收用户拍摄的菜单图片（Base64 编码）和一组健康标签（如"花生过敏""低糖""素食"等）。核心处理逻辑在 `NutriVisionClient.analyze_menu()` 方法中：首先预处理 Base64 数据，剥离可能存在的 `data:image/xxx;base64,` 前缀；然后构造多模态 prompt，要求 Gemini 2.0 Flash 模型扮演"专业数字营养师"角色，识别图片中的每道菜品并给出名称、估算热量、食材列表和过敏原警告，同时根据用户的健康标签推荐 Top 3 最适合的菜品，最后生成一段综合性的健康饮食建议。

为了应对 AI 模型返回格式不一致的问题，系统实现了防御性的 `_standardize_response()` 方法，能兼容 `items`、`dishes`、`menu_items` 等不同字段命名习惯，并自动将食材字段从逗号分隔的字符串转换为列表格式。前端收到 AI 分析结果后，在 NutriVisionResultScreen 页面中以卡片化界面展示每道菜品的营养信息，过敏原信息以醒目的警告标签呈现。

## 五、营销与优惠券系统的实施

营销服务的优惠券计算流程从 `CouponCalculationService` 开始：首先查询当前用户所有可用且未过期的优惠券，加载对应的优惠券模板信息，校验每张券的使用门槛条件（如满减券的最低消费金额），然后调用 `CouponCombinationService` 的组合优化算法。组合算法首先将所有可用券按照互斥关系分为两组——如果一张券的 exclusiveIds 字段中包含另一张券的模板 ID，则两者互斥。互斥性校验采用双向检查，即 A 排斥 B 和 B 排斥 A 都会被正确识别。分组完成后，算法通过位掩码（bitmask）枚举可叠加券的所有 $2^n$ 个子集，再将每个枚举出的可叠加组合分别与每张互斥券搭配，最终选择总优惠金额最大的一个方案返回。每种优惠券的优惠金额计算方式各不相同：折扣券按订单金额乘以折扣率计算差价，满减券直接减去固定金额，无门槛券减去面值（不低于零），免运费券减去运费金额。

智能发券引擎基于 `auto_issuance_rules` 表中的规则定义工作。每条规则指定了触发类型、关联的优惠券模板、触发频率（ONCE/DAILY/WEEKLY/MONTHLY）、优先级、总发放限额和单用户限额。当系统检测到触发事件（如新用户注册、用户信用等级从 3 升到 4）时，引擎查找匹配的活跃规则，校验发放限额后自动为用户创建优惠券记录。发放历史记录在 `auto_issuance_history` 表中，包含触发时间、是否成功和失败原因，便于运营人员追踪发券效果。

## 六、平台运营与结算系统的实施

平台服务的佣金计算由 `CommissionService.calculateCommission()` 方法实现，该方法在每笔订单支付确认后被调用。它首先查询目标商家所有处于活跃状态的按单（PER_ORDER）计费订阅，然后遍历每个订阅，根据服务的费率类型计算该服务应收取的佣金金额：PERCENTAGE 类型用订单金额乘以费率值，FIXED_PER_ORDER 类型取固定金额，FIXED_MONTHLY 类型不计入单笔订单的佣金。每条佣金记录关联到具体的订单 ID、商家 ID 和服务 ID，状态为 PENDING（待结算）。当订单发生退款时，`refundCommission()` 方法将该订单的所有佣金记录状态更新为 REFUNDED。

结算单的生成逻辑在 `SettlementService.generateSettlements()` 方法中，长达 683 行代码。该方法首先查询所有存在 PENDING 状态佣金记录的商家，然后为每个商家统计待结算期间内的订单总额（GMV）、总佣金金额和净收入（GMV 减去佣金），创建结算单记录并将相关的佣金记录关联到该结算单上。结算单号的格式设计为 `ST{年月}{W或M}{商家ID末3位}{序号}`，例如 `ST202602W0010001` 表示 2026 年 2 月的周结算单、商家编号末三位为 001、序号 0001。

结算单的状态流转设计为：PENDING_CONFIRM（待商家确认）→ CONFIRMED（商家已确认）或 DISPUTED（商家有异议）→ PAID（已打款）或 CANCELLED（已取消）。系统实现了超时自动确认机制：`autoConfirmExpiredSettlements()` 方法定时检查所有超过确认期限（默认 3 天）仍处于 PENDING_CONFIRM 状态的结算单，将其自动确认。管理员可以在 Web 管理端查看结算趋势分析（支持按日、周、月、年维度聚合），处理商家的异议申诉，执行批量打款操作。

## 七、用户系统与信用等级的实施

用户服务的信用等级系统通过 `CreditService` 实现，信用等级从 1 到 5 级，新注册用户默认 5 级（最高信用）。降级规则为：如果用户在最近 7 天内取消订单达到 3 次及以上，信用等级下降 1 级，最低降至 1 级。升级规则为：如果用户最近 7 天内没有任何取消记录，并且最近 30 天内累计完成了 3 笔以上订单，信用等级上升 1 级，最高升至 5 级。信用等级的变动与营销系统的智能发券引擎联动——信用升级事件可以触发 CREDIT_UPGRADE 类型的自动发券规则，奖励用户一张优惠券作为信用提升的激励。

## 八、前端应用的实施

移动端前端使用 React Native 构建，主界面 HomeScreen.tsx 是代码量最大的单文件（1357 行），集成了推荐餐厅列表、语音搜索、NutriVision 拍照分析、天气感知提示、健康上下文显示和开发者调试面板等核心功能。前端通过统一的 `serviceConfig.js` 配置文件管理所有后端服务的地址，根据运行平台（Android/iOS）和运行环境（开发/生产）自动切换 API 地址。

前端的服务层（services 目录）封装了与各后端微服务的通信逻辑。`recommendationService.js` 实现了 V2 推荐请求（构建包含 health_context 和 weather_context 的请求体调用 `/agents/recommend`）和端云协同请求（调用 `/agents/edge-synergy-recommend`，处理 NO_MATCH 降级），并实现了自动餐厅导入功能——当推荐服务返回的 POI 搜索结果中包含平台未收录的餐厅时，前端会异步调用商家服务的接口将其导入。`weatherService.ts` 调用后端 MCP 工具接口获取天气分析数据，并维护 10 分钟的本地缓存避免频繁请求。`aiPricingService.js` 封装了定价提案管理的 7 个 API 端点，支持商家查看待审批提案、历史提案，批准或拒绝提案，开关自动审批功能以及手动触发 AI 分析。

移动端同样包含完整的商家端页面，在 `screens/merchant/` 目录下实现了商家仪表盘、入驻流程、店铺信息编辑、菜单管理、智能定价管理、退款审核、增值服务市场和结算管理等八个核心页面。Web 管理端在 `src_frontend_web/` 目录下，覆盖了管理员所需的全部功能页面，包括数据看板、营销管理、商家管理、订单管理、用户管理、信用管理、平台服务管理、结算管理、佣金查询和系统监控等。

## 九、可观测性与 DevOps 的实施

系统的可观测性方案采用 Prometheus + Grafana + Zipkin 三位一体架构。所有 Java 微服务通过 Spring Boot Actuator 和 Micrometer 暴露 Prometheus 格式的 metrics 端点（`/actuator/prometheus`），全局配置文件 `application.yml` 中统一开启了 health、info、metrics、prometheus 端点的暴露。Prometheus 定期抓取各服务的 metrics 数据，Grafana 提供可视化仪表盘用于实时监控各服务的请求量、错误率和延迟分布。Zipkin 作为分布式链路追踪系统，记录跨服务调用链的完整路径和耗时，帮助定位微服务间的性能瓶颈。所有这些监控基础设施都通过 Docker Compose 的 enhanced profile 按需启动，不会在日常开发时消耗额外的系统资源。
