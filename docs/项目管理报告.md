# FoodMate-AI 项目管理报告

## 一、项目范围与目标

### 1.1 产品简述与目标

FoodMate-AI 定位为下一代基于多智能体协作（Multi-Agent System）与端云协同架构的智能外卖平台。本项目旨在解决传统外卖平台推荐系统缺乏环境感知能力、定价机制僵化以及健康指导功能缺失的痛点。产品的核心价值主张在于利用 AI 技术重构外卖服务的全链路体验，从“人找饭”进化为“饭找人”的智能主动服务模式。

核心目标主要体现在三个层面：

首先，构建高精度的**场景化智能推荐系统**。该系统基于 `ReAct` (Reason+Act) 范式的多智能体架构，创新性地引入了环境感知层。推荐引擎不仅仅依赖用户的历史订单数据，更深入理解用户所处的实时环境上下文（Context），包括当前的天气状况（如雨雪天推荐热食）、交通拥堵程度（避免推荐远距离商家）、当前时间段（工作日午餐推荐快餐，周末晚餐推荐大餐）以及用户的即时健康需求（如“低糖”、“高蛋白”）。通过这些多维度的感知，系统能够提供比传统协同过滤更精准、更具人性化的个性化推荐。

其次，实施**动态定价赋能**策略。通过集成的 `APScheduler` 定时任务与事件驱动机制，系统能够实时监控市场供需关系。定价引擎会基于历史销量、当前库存、竞争对手价格以及天气交通等外部因素，利用强化学习算法自动调整菜品价格。例如，在暴雨天气导致运力紧张时，适当提高配送费或热门菜品价格以平衡需求；在非高峰时段，对库存积压的菜品进行自动打折促销。这种动态定价机制不仅能帮助商家最大化收益，也能通过价格杠杆有效调节市场供需平衡。

最后，提供**多模态健康指导**服务。利用先进的 AI 视觉技术（基于 Gemini 1.5 Flash 模型），用户只需拍摄菜单或食物照片，系统即可实现毫秒级的菜品识别与营养分析。配合端侧大模型（Qwen-2.5-0.5B），系统能在保护用户隐私的前提下，提供即时的健康饮食建议。例如，通过分析照片中的菜品成分，提醒用户潜在的过敏原，计算卡路里摄入量，并根据用户的健康档案（如糖尿病患者、健身人群）给出个性化的食用建议。

### 1.2 技术架构与选型

本项目在技术架构上精心设计，采用了前后端分离的现代化微服务架构，确保了系统的高可用性、可扩展性以及各模块的独立演进能力。

**前端交互层**基于 React Native 0.83.1 框架构建，支持 Android 与 iOS 双端运行，实现了“一次编写，处处运行”的高效开发模式。在端侧 AI 能力的构建上，项目展现了极高的技术壁垒：
*   集成了 `react-native-vosk` 实现完全离线的语音识别。通过预置的 Vosk Small Model，App 能够在无网络环境下精准识别用户的语音指令，且所有语音数据均在本地处理，绝不上传云端，彻底解决了用户对隐私泄露的担忧。
*   集成了 `llama.rn` 运行 Qwen-2.5-0.5B 量化模型。这使得移动端具备了强大的本地自然语言理解能力，能够处理敏感指令（如涉及个人健康数据的查询）并进行意图识别，无需依赖云端大模型，进一步降低了延迟并提升了数据安全性。
*   视觉交互方面，采用 `react-native-image-picker` 捕获菜单图像。针对移动网络不稳定的特点，前端实现了智能的图片压缩算法，将图片转换为高效的 Base64 格式后上传，既保证了识别精度，又大幅减少了流量消耗。

**后端业务层**以 Java Spring Boot 3.2.0 为核心框架，结合 Spring Cloud 2023.0.0 生态体系，构建了稳固的微服务底座。
*   服务治理依赖 Eureka Server (8761) 作为注册中心，实现了服务实例的自动注册与发现。
*   服务间通信利用 OpenFeign 实现声明式的远程调用，简化了代码逻辑。
*   为了防止服务雪崩，引入了 Resilience4j 组件，为关键服务接口配置了熔断器、在调用失败时自动降级，并实施了限流保护。
*   微服务矩阵分工明确：`merchant-service` (8081) 专注解决商家入驻、店铺管理及菜单维护的复杂业务逻辑；`marketing-service` (8082) 负责构建灵活的优惠券系统与营销活动引擎；`order-service` (8084) 作为核心交易引擎，全权管理订单从创建、支付、履约到售后的全生命周期状态流转。

**AI 智能层**作为系统的"大脑"，采用 Python FastAPI 构建，充分利用其异步高并发特性处理复杂的 AI 推理任务。
*   推荐服务 (`recommendation-service`) 深度整合了 LangChain 0.1.0 与 LangGraph 框架，构建了包含 Information Gatherer（信息收集）、Reasoner（推理决策）和 Executor（执行反馈）的智能体工作流。底层算法融合了 LightGBM 的快速特征排序与 DeepFM 的深度语义匹配，并接入 DeepSeek-Chat 或 OpenAI API 进行基于 `ReAct` 模式的高级推理，确保推荐结果既符合用户偏好又具备逻辑解释性。
*   定价服务 (`ai-pricing-service`) 拥有独立的逻辑闭环。它基于 `APScheduler` 调度器执行周期性的（如每小时或更细粒度）定价分析任务。同时，利用 `aio-pika` 异步监听 RabbitMQ 中的 `orders.payment.completed` 等关键业务事件。一旦发生交易，系统会立即触发实时价格微调算法，根据最新的销量波动动态调整价格策略。
*   视觉服务 (`nutrivision-service`) 对接 Google Gemini 1.5 Flash 模型，利用其强大的多模态理解能力。服务内部实现了完善的图片预处理机制，包括 5MB+ 图片大小预检、格式校验以及异常熔断机制，确保在高并发图片上传场景下的服务稳定性。

**数据基础设施**方面，项目选用了多模态数据库存储策略以适应不同类型的数据需求。
*   PostgreSQL 15 作为核心关系型数据库，存储结构化业务数据。通过分库设计（`food_delivery_db` 与 `ai_pricing_db`），实现了业务交易数据与定价策略数据的物理隔离，避免了单一数据库的性能瓶颈。
*   MongoDB 6.0 用于存储非结构化、多变的用户画像数据（如用户的浏览轨迹、点击热力图、复杂的健康标签等），利用其灵活的文档模型支持快速迭代。
*   Redis 7.0 承担热点数据缓存的重任，缓存热门菜单、用户信息及会话状态，极大降低了数据库的读取压力。
*   RabbitMQ 3.12 作为消息中间件，实现了服务间的异步解耦。核心交易链路（如下单成功触发库存扣减、支付成功触发营销发券）均通过消息队列异步处理，保证了系统的高吞吐量与最终一致性。
*   所有服务均通过 Docker Compose 进行统一编排，每个容器都配置了严格的 `healthcheck` 机制（如 `pg_isready` 检测数据库端口连通性），确保在服务启动时，依赖组件已完全就绪，避免了分布式系统常见的启动竞态问题。

### 1.3 工作分解结构 (WBS)

项目全生命周期被精细拆解为六个核心阶段，每个阶段均有明确的产出物与里程碑，确保项目按部就班、高质量推进。

**第一阶段：规划与设计 (Planning & Design)**
此阶段重点在于夯实项目基础，从抽象的想法转化为具体的工程蓝图。
*   **需求工程**: 团队不仅编写了详尽的用户故事，还通过 Figma 完成了高保真的 UI/UX 原型设计，涵盖了App主要交互流程的20+个页面。需求规格说明书（SRS）详细定义了功能性与非功能性需求。
*   **架构设计**: 技术团队完成了微服务的详细拆分规划，绘制了系统架构图与数据流图。数据库设计方面，完成了包含 27 张表的 ER 模型设计，详细定义了表间关系、索引策略及外键约束。API 接口定义采用 OpenAPI (Swagger) 标准，输出了包含请求/响应示例的完整接口文档。
*   **技术预研**: 针对项目中的高风险技术点进行了 PoC 验证。重点验证了 `llama.rn` 在红米 Note系列等中低端机型上的端侧推理性能，以及 RabbitMQ 在高并发场景下的消息可靠性与延迟表现。

**第二阶段：基础设施与核心服务**
该阶段聚焦于 DevOps 环境搭建与 Java 基础服务的开发，筑牢系统底座。
*   **DevOps**: 通过 Docker Compose 编写了详细的编排文件 `docker-compose.dev.yml`，完成了 PostgreSQL、MongoDB、Redis 及 RabbitMQ 中间件的容器化部署与网络配置。
*   **Java 基础服务**: 开发了 `user-service`，实现了基于 JWT 的无状态鉴权机制与用户注册登录流程；`merchant-service` 完成了商家入驻申请、店铺信息维护及复杂的菜单管理（支持多规格、多属性）功能。抽取了 `food-platform-common` 模块，封装了统一的 API 响应结构 (`ApiResponse<T>`)、全局异常处理 (`GlobalExceptionHandler`) 及常用工具类，避免了代码重复。
*   **中间件集成**: Spring Cloud Eureka 注册中心成功部署，实现了微服务的自动发现。Config 配置中心集成了 Git 仓库，实现了配置文件的集中管理与动态刷新。OpenFeign 客户端被广泛集成到各服务中，实现了服务间像调用本地方法一样便捷的远程通信。

**第三阶段：业务服务开发**
本阶段致力于核心业务逻辑的实现，是项目开发工作量最大的部分。
*   **订单核心**: `order-service` 实现了极其复杂的订单状态机，涵盖了从 `CREATED` (创建)、`PAID` (已支付)、`CONFIRMED` (商家接单)、`DELIVERING` (配送中) 到 `COMPLETED` (完成) 的全流程状态流转逻辑，并处理了各种异常状态（如超时自动取消、拒单退款）。
*   **营销系统**: `marketing-service` 构建了灵活的优惠券模板系统，支持满减、折扣、免运费等多种类型。智能发券规则引擎能够根据用户标签（如新用户、沉睡用户）自动发放定向优惠券。
*   **运营平台**: `platform-service` 完成了结算单生成逻辑，能够根据订单金额、平台抽成比例及营销分摊规则，自动生成商家的每日/每周结算单，并实现了商家对账流程。
*   **画像服务**: `profile-service` 建立了基于 MongoDB 的用户标签体系，能够实时采集用户的浏览、搜索、下单等行为日志，并进行清洗入库，为后续的推荐算法提供数据支撑。
*   **消息驱动**: 完成了 RabbitMQ 的 Exchange (Topic模式) 与 Queue 的详细配置，确保了 `order.paid`、`order.created` 等关键业务事件能够被准确路由到对应的消费者，实现了各业务模块的解耦。

**第四阶段：AI 服务与算法攻关**
此阶段是项目的技术攻坚期，也是体现项目创新点的关键环节。
*   **推荐与编排**: 推荐团队开发了 `recommendation-service`，利用 LangGraph 编排了 Environment（环境感知）、Profile（画像分析）、Decision（决策生成）的多智能体工作流。实现了 Contextual Bandit 算法，能够在探索新口味与利用已知偏好之间取得平衡。
*   **动态定价**: 定价团队完成了 `ai-pricing-service` 的开发。构建了基于 `sales_history` 的数据采集管线，能够实时聚合各菜品的销量数据。APScheduler 定时任务能够按预定策略触发定价分析，Gemini 定价策略 Prompt 经过多轮调优，能够输出合理的调价建议。
*   **视觉分析**: 视觉团队专注于 `nutrivision-service` 的 Prompt 工程，通过 Few-Shot Learning (少样本学习) 提升了 Gemini 模型对菜品成分识别的准确率。完善了图片上传的异常熔断机制，防止恶意攻击或大文件传输导致服务崩溃。
*   **外部 API**: 完成了和风天气与高德地图 API 的深度集成，封装了统一的调用客户端，并实现了详细的降级策略（如 API 超时时使用默认环境参数），保证了主流程的健壮性。

**第五阶段：前端与移动端开发**
移动端开发全面铺开，将后端服务能力转化为具体的用户界面。
*   **React Native App**: 完成了消费者端的首页推荐流（支持下拉刷新、无限加载）、语音点餐悬浮球、购物车逻辑、订单实时追踪（对接地图组件）、个人中心等核心模块。商家端则实现了订单语音播报、菜单上下架管理、经营数据看板（图表展示）、定价提案审批流等功能。
*   **端侧 AI**: 这里的亮点是将 Vosk 离线语音引擎集成到 App 中，实现了毫秒级的语音唤醒与识别。同时，成功将 Qwen-0.5B 大模型量化并部署到端侧，通过 `llama.rn` 接口实现了本地化的自然语言指令解析，提升了交互的流畅度与智能化。
*   **Web 管理台**: 开发了基于 React 的平台管理员后台，提供了商家入驻审批、全平台订单监控、营销活动配置等功能，方便运营人员进行日常管理。

**第六阶段：测试与交付**
项目收尾阶段，重点在于全面的质量保障与成果交付。
*   **自动化测试**: 建立了完善的自动化测试体系。Java 后端采用 JUnit 5 + Mockito 编写了大量的单元测试，覆盖了核心业务逻辑。Python 算法服务采用 pytest 框架进行测试。针对关键 API 接口，编写了 Postman Collection 脚本并集成到 Newman 中进行自动化回归测试。
*   **系统联调**: 进行了全链路集成测试，模拟真实用户场景（从语音点餐到支付、履约）验证各服务间的协作是否顺畅。使用 k6 工具进行了性能压测，重点测试了高并发下单与推荐接口的响应时间，并根据压测结果进行了性能调优。
*   **AI 效果评估**: 通过模拟 A/B Test 的方式，对比了不同定价策略与推荐算法的效果，收集数据评估 AI 模型的实际业务价值。
*   **文档与答辩**: 整理并交付了全套项目文档，包括详细的项目管理报告、技术架构文档、API 接口文档、部署运维手册等。录制了系统功能演示视频，为最终的项目答辩做好了充分准备。

上表详细界定了项目核心任务的所有权与协作关系。在微服务开发领域，黄弋涵主要通过 *Service Domain Ownership* 的方式，主导了用户、商家及订单等高频交易链路服务的建设，确保了核心交易系统的稳定性与数据一致性；而关镜文则侧重于营销与平台运营等支撑性服务，构建了灵活的活动配置与清结算能力。在 AI 与算法层面，黄弋涵专注于视觉与定价的垂直领域应用，利用特定的模型解决具体业务痛点；关镜文则负责构建通用的推荐算法底座与 MLOps 流程，为上层应用提供统一的智能决策支持。基础设施与移动端开发主要由关镜文承担，这确保了端到端的用户体验与底层运维架构的一致性。这种 "T型" 人才互补结构，既保证了每个模块有专人深度负责 (R/A)，又通过咨询 (C) 与知情 (I) 机制促进了跨领域的知识共享与风险共担，最大化了两人团队的交付效能。

## 三、资源管理

### 3.1 实物与API资源

在大模型 API 资源配置上，项目采取了精细化的分级路由策略，以在有限的预算下平衡成本与性能。**主力模型**选用 DeepSeek V3（配置为 OpenAI 兼容接口模式），主要承担推荐理由生成、复杂逻辑推理及长文本处理任务。我们看重其主要原因在于 DeepSeek 极具竞争力的 Token 价格与其强大的中文语境理解能力，能显著降低高频推荐请求的边际成本。**视觉模型**则指定由 Google Gemini 1.5 Flash 承接，该模型在多模态理解方面表现优异，且拥有较高的免费调用额度与快速响应特性，完美适配 `nutrivision-service` 对菜单图片进行实时识别与结构化提取的需求。此外，为了防止单一供应商故障，我们配置了 OpenAI GPT-3.5/4o-mini 作为系统的 **Fallback 热备方案**。一旦监控系统检测到 DeepSeek 服务不可用或响应超时，流量将自动无缝切换至 OpenAI 通道，确保核心业务连续性。

**端侧计算资源**的规划充分考虑了设备兼容性与隐私保护的需求。**Android 设备**作为主要的端侧推理载体，我们明确了最低硬件规格：需支持 Android 10+ 操作系统以兼容最新的 ML API，且内存需达到 6GB+ 以保证 `llama.rn` 加载 Qwen-2.5-0.5B 量化模型（INT4 或 INT8 量化）时不会因内存溢出而 Crash。对于语音交互功能，项目在 App 发布包中预置了约 50MB 大小的 Vosk Small Model (CN)，这是一个经过剪枝优化的轻量级声学模型，实现了在无网络环境下的高效离线语音指令解析，将语音识别延迟控制在 500ms 以内。

**服务设施**方面，外部数据源的接入对于增强 AI 的上下文感知能力至关重要。项目深度集成了 **和风天气 API**，用于实时获取精确到区县级的气温、降水概率及气象预警信息。这些数据不仅直接影响推荐系统的加权逻辑（如雨天增加火锅、汤类权重），还通过规则引擎触发配送费的动态调整。同时，接入 **高德地图 API** 获取实时交通拥堵系数（TPI），为动态定价算法提供关键的配送成本参数。我们还使用了其路径规划接口来估算更准确的配送预达时间（ETA），提升用户预期管理。

### 3.2 协作工具

团队采用了一套标准化的协作工具链来提升研发效率与过程透明度。**代码管理**基于 GitHub 平台，严格遵循 Git Flow 工作流。Feature 分支是日常开发的主战场，所有的代码合并必须通过 Pull Request (PR) 流程，这强制执行了代码审查（Code Review）机制，并在 Approve 后依次合并入 Develop（开发集成环境）与 Main（生产发布环境）分支，确保主干代码的稳定性。**接口文档**管理实现了高度自动化，Java 微服务利用 SpringDoc OpenApi 自动生成 Swagger UI 在线文档，确保文档与代码实时同步。配合 Postman Collection 进行接口调试与共享，并在 Postman 中维护了全套的环境变量（Environment Variables），便于在本地、开发、生产环境间快速切换。**数据库管理**工具多样化，Mongo Express 提供了 Web 端的 MongoDB 可视化管理能力，便于快速查看 JSON 文档结构；而 PostgreSQL 的日常维护、索引优化分析（Explain Analyze）则依赖于 pgAdmin 或 DBeaver 等专业客户端工具。

### 4.3 依赖风险

外部 API 的高可用性是系统稳定运行的关键挑战，特别是 DeepSeek V3 推理服务和 Gemini 1.5 Pro 视觉服务的稳定性。

*   **API 熔断策略**:
    *   **配置参数**: 针对所有外部大模型调用，配置 Resilience4j 断路器。
    *   **失败率阈值**: 当 50% 的请求在滑动窗口大小为 10 的范围内失败时，熔断器打开。
    *   **半开状态等待**: 熔断后等待 60 秒进入半开状态，允许 3 个测试请求通过。
    *   **半开调用限制**: 半开状态允许 3 次请求探测，若成功则关闭熔断器。
*   **超时控制**:
    *   **连接超时**: 设置为 2000 毫秒以防止 TCP 握手阻塞。
    *   **读取超时**: 设置为 15000 毫秒，因为大模型推理通常耗时较长。一旦超时立即抛出超时异常并触发降级。
*   **降级机制**:
    *   **一级降级**: 主模型 DeepSeek 失败时，自动切换至备用模型 Gemini Flash，通过策略模式设计实现无缝切换。
    *   **二级降级**: 若所有在线模型不可用，系统将返回模拟数据或缓存的静态推荐结果，确保前端不白屏。
*   **消息队列堆积风险**:
    *   **RabbitMQ 配置**: 订单状态队列设置了 60000 毫秒的消息过期时间和死信交换机。
    *   **死信处理**: 消费失败的消息会被路由到死信队列，由专门的补偿任务每 5 分钟轮询处理一次，进行人工介入或重试。

## 五、质量保证与控制

### 5.1 质量评估标准

基于 McCall 软件质量模型，我们为项目设定了严格且可量化的质量标准，重点关注代码覆盖率、性能指标和安全基线。

**正确性**:
*   **单元测试**: 核心业务模块如订单和支付的 JUnit 5 测试覆盖率必须超过 85%。使用 Jacoco 插件生成覆盖率报告，低于阈值直接构建失败。
*   **集成测试**: 关键 API 如创建订单接口必须通过 TestContainers 启动真实的 PostgreSQL 容器进行测试，断言数据库状态变更，通过率需达到 100%。
*   **Bug 修复标准**: Bug 重开率控制在 5% 以内，这意味着修复必须经过开发者自测和 QA 验证。

**效率**:
*   **移动端性能**: App 首页首屏可交互时间需控制在 1.2 秒以内。冷启动时间需控制在 2 秒以内。使用 React 的记忆化组件和回调钩子优化组件渲染，减少不必要的重渲染。
*   **服务端响应**: 推荐接口的 P95 响应时间需低于 300 毫秒。Redis 缓存命中率应超过 90%，并且使用布隆过滤器防止缓存穿透。
*   **AI 推理**: 端侧 Qwen 模型首次 Token 生成时间需低于 800 毫秒，语音识别延迟需低于 500 毫秒。

**可靠性**:
*   **服务等级协议**: API 服务可用性目标为 99.9%。
*   **数据一致性**: RabbitMQ 开启发布者确认和消费者手动确认。订单支付成功后，确保 100% 触发订单已支付事件，库存扣减采用尝试确认取消柔性事务或基于版本号的乐观锁。
*   **故障恢复**: 平均修复时间需控制在 1 小时以内。Docker Swarm 或 Kubernetes 配置存活探针和就绪探针，每 10 秒检测一次健康检查端点，失败自动重启容器。

**安全性**:
*   **漏洞扫描**: 项目必须通过 OWASP ZAP 的前十大漏洞扫描，例如 SQL 注入和跨站脚本攻击。
*   **加密传输**: 所有 HTTP 通信强制升级为 HTTPS 协议。敏感数据如密码使用 BCrypt 哈希算法存储。
*   **鉴权**: JWT 令牌签名使用 HMAC SHA256 算法，密钥长度至少 32 字符，令牌有效期 1 小时，刷新令牌有效期 7 天。

### 5.2 质量控制技术流

我们实施了多层次、全流程的持续集成与持续部署质量保障体系。

**1. 开发阶段**:
*   **IDE 插件**: 强制安装 SonarLint 实时检测代码异味。
*   **预提交钩子**: 使用 Husky 和 Maven Checkstyle 插件。
    *   **代码风格**: 遵循 Google Java 风格指南。
    *   **前端规范**: 遵循 Airbnb React 配置。
    *   **格式化**: 使用 Prettier 统一格式。
    *   任何代码规范错误或格式问题将阻止 Git 提交。

**2. 构建与合并阶段**:
*   **持续集成流水线**: 每次推送触发。
    *   **步骤一**: 运行 mvn clean test 执行 120 多个单元测试。
    *   **步骤二**: 运行 npm run test 执行 Jest 组件快照测试。
    *   **步骤三**: SonarQube 扫描。设置质量门禁为新代码覆盖率大于 80% 且无严重问题。不达标直接阻止合并请求。
*   **代码审查**: GitHub 合并请求强制要求至少 1 名资深开发者批准。不仅检查逻辑，还要检查 SQL 性能例如是否使用了索引。

**3. 测试与部署阶段**:
*   **性能测试**: 使用 K6 脚本模拟 50 个虚拟用户并发访问推荐接口，持续 5 分钟，通过 Jenkins 定时触发。
    *   检查响应状态码是否为 200 且响应时间是否小于 500 毫秒。
*   **端侧兼容性**: 利用 AWS Device Farm 或 Firebase Test Lab，在 Android 10 至 14 的 10 款主流机型上运行 Appium 自动化 UI 测试。
*   **灰度发布**: 新版算法上线时，通过 Nginx 权重配置，仅将 5% 的流量引入新服务，观察错误率和延迟，无异常后逐步全量。

## 六、监督与控制

### 6.1 进度跟踪机制

项目采用标准的 Scrum 敏捷流程作为管理核心，通过严格的迭代周期和过程仪式确保交付质量。我们将 Sprint 周期设定为高度紧凑的 2 周（即 10 个工作日），并利用 GitHub Projects 的自动化看板作为单一事实来源。在看板的具体实践中，我们严格定义了各泳道的流转规则：所有进入“准备开发”状态的任务必须具备明确的验收标准（Acceptance Criteria），处于“进行中”的任务受到在制品限制（WIP Limit 固定为 3），以防止开发人员任务切换过于频繁。任何任务要流转至“已完成”，必须不仅通过自动化集成测试，还需经过产品负责人的最终 UI 验收。这种端到端的看板管理，确保了从需求提出到功能交付的全流程可视化。

为了应对项目执行中的细节偏差，我们建立了一套高频短周期的反馈闭环。每日上午 09:30 至 09:45 举行的站立会议被严格限制在 15 分钟以内，所有成员通过钉钉或 Slack 等即时通讯工具提前 5 分钟接收机器人推送的昨夜 Git 提交记录摘要，从而将会上时间聚焦于真正的阻碍（Blocker）排除，而非流水账式的进度汇报。同时，我们通过每日更新的燃尽图量化风险，一旦检测到剩余任务点数偏离理想曲线超过 15% 的阈值，Scrum Master 将立即启动干预机制，包括但不限于移除低优先级的用户故事或临时增加关键模块的结对编程资源，确保迭代目标的达成。

每双周五下午的迭代回顾会议（Sprint Retrospective）则是团队持续改进的关键时刻。我们采用“开始-停止-继续”（Start-Stop-Continue）模型进行结构化复盘，分析本周期内的技术债务与协作摩擦。复盘的产出物不仅仅是会议记录，而是必须落实到具体的行动项（Action Items）并在 Jira 中生成对应的高优先级任务，例如“下周需将订单服务的数据库连接池从 Commons DBCP 迁移至 HikariCP 并完成压力测试”。通过这种“复盘即行动”的机制，我们将每一次的问题转化为下一次迭代的效能提升。

### 6.2 变更与配置管理

版本控制策略依托于严谨的 Git Flow 工作流，从根源上保障了代码库的稳定性与可追溯性。生产环境分支 Main 被配置为受保护状态，禁止任何形式的直接推送，所有的代码合入必须通过 Pull Request 机制，并强制要求通过 GitHub Actions 持续集成流水线的自动化检查。日常的功能开发均在独立的 Feature 分支进行，开发完成后通过 Rebase 策略合并回 Develop 主干，以保持提交历史的线性整洁。针对线上可能出现的紧急故障，我们启用了 Hotfix 分支流程，修复代码经本地验证后，会同时 Cherry-pick 至主干与开发分支，确保不同环境下的版本逻辑保持严格一致。

在配置管理层面，项目采用了环境隔离与分层管理的综合策略来确保系统的灵活性与安全性。所有的微服务通用配置，如 Jackson 序列化策略、日志级别定义以及 Actuator 端点暴露规则，都被集中管理在 `backend/config-repo/application.yml` 文件中，这种做法极大地减少了重复配置带来的维护成本，并确保了所有服务在基础行为上的一致性。对于数据库连接凭证、第三方 API 密钥（如以及  Vosk 模型路径）等敏感信息，我们坚决摒弃了硬编码的做法，而是通过 `.env` 环境文件结合 Docker Compose 的变量注入机制，在容器启动时动态加载到运行时环境中，从而在代码仓库泄露的情况下也能有效保护核心凭据的安全。

针对数据库 Schema 的演进，我们实施了严格的版本化脚本管理机制。所有的数据库变更不再依赖口头沟通或手动操作，而是必须转化为 `backend/scripts/` 目录下的 SQL 迁移脚本。目前项目已经维护了从基础架构 `01_schema.sql` 到最新的智能定价表 `15_smart_issuance_tables.sql` 的完整演进历史，涵盖了核心交易表结构、AI 定价服务专用表以及初始化种子数据。每次进行系统部署前，运维人员需按序列号顺序执行未应用的 SQL 脚本，这种机制确保了数据库结构与应用代码版本的严格同步，避免了因 Schema 不一致导致的运行时错误。

### 6.3 成本与资源监控

为了在有限的计算资源下最大限度地保障系统运行的稳定性，我们采取了务实且具有针对性的监控与配额策略。在资源监控技术栈的选择上，项目预置了基于 Prometheus 与 Grafana 的轻量级可观测性方案。各微服务通过集成 Spring Boot Actuator 组件，主动暴露了标准的 Prometheus 监控端点，实时输出 JVM 内存使用量、垃圾回收频率以及 HTTP 请求延时等核心性能指标。虽然目前尚未配置复杂的自动化告警规则，但通过 Grafana 可视化看板，团队已经能够直观地查看到服务运行状态与 CPU 水位趋势，为后续的性能调优与瓶颈分析提供了坚实的数据支撑。同时，针对微服务链路调用复杂的特点，我们集成了 Zipkin 组件，通过 Spring Cloud Sleuth 自动注入 Trace ID，实现了跨服务调用的全链路追踪，便于快速定位请求延迟高或调用失败的具体环节。

在成本控制方面，鉴于 AI 推理服务（DeepSeek 与 Gemini）的高昂调用成本，我们目前主要在应用层实施了严格的限流与防失控策略。在 `agent_orchestrator.py` 等 AI 交互的核心代码中，我们硬性设置了 `max_tokens` 参数（通常限制为 2000），这一简单的配置有效防止了模型生成过长回复导致的 Token 消耗失控风险。对于 API 的调用额度管理，由于暂未集成自动化的成本仪表盘，目前的监控主要依赖于定期查阅云服务商后台的账单与用量统计，并结合服务日志中的调用记录进行人工核对。此外，为了防止因第三方 API 额度耗尽导致服务完全不可用，系统设计了基于 Resilience4j 的熔断降级逻辑，当检测到 AI 服务连续调用失败或响应超时，将自动切换至备用逻辑，如返回缓存的推荐结果或静态规则，确保核心业务不受影响。
