# 2 相关工作

本章围绕项目涉及的五个核心技术方向，系统梳理近三年（2023–2025）内最具代表性的已有技术方案，分析其技术路线、核心贡献与局限性，并阐明本项目 FoodMate-AI 在此基础上的改进与超越之处。

---

## 2.1 基于大语言模型的多智能体推荐系统

### 2.1.1 RecAgent：基于 LLM Agent 的用户行为模拟推荐框架

**来源**：Wang L, Zhang J, Yang H 等. *Agent-based User Behavior Simulation with Large Language Model based Agents*. ACM Transactions on Information Systems (TOIS), Vol. 43, Issue 2, 2025. (arXiv: 2306.02552)

**技术路线**：RecAgent 将推荐系统中的用户行为建模为 LLM 驱动的自主智能体行为。每个用户被建模为一个 Agent，内部包含三个模块：画像模块（Profiling，赋予不同角色特征）、记忆模块（Memory，建模行为动态演化）和行动模块（Action，模拟点击、浏览、聊天等行为）。通过多 Agent 的自主交互模拟，RecAgent 可以生成逼真的用户行为数据用于推荐算法的离线评估，在用户行为预测任务上超越最佳基线方法约 45.8%。

**局限性**：RecAgent 定位于推荐系统的**离线模拟与评估工具**，而非面向真实用户的在线推荐引擎。其 Agent 的核心目标是"模拟用户怎么想、怎么做"，而非"感知环境并做出推荐决策"。此外，RecAgent 不涉及任何环境上下文信息（天气、交通、时段）和健康数据的融合，也不具备在线学习能力。

**本项目的改进**：FoodMate-AI 将 Agent 范式从"模拟用户"转向"服务用户"——三个智能体（ContextAgent、ProfilerAgent、DecisionAgent）各自负责一个维度的信息感知与分析，通过 LangGraph 状态图引擎进行有条件的串行/并行编排，最终由 DecisionAgent 的 Contextual Bandit 算法做出实时排序决策。与 RecAgent 的离线模拟不同，FoodMate-AI 是一个**面向生产环境的在线推荐系统**，支持 50+ 维上下文信号的实时融合和基于用户反馈的在线学习。

### 2.1.2 AgentCF：基于自主语言智能体的协同过滤

**来源**：Zhang J, Hou Y, Xie R 等. *AgentCF: Collaborative Learning with Autonomous Language Agents for Recommender Systems*. Proceedings of the ACM Web Conference 2024 (WWW '24), Singapore, May 2024. (arXiv: 2310.09233)

**技术路线**：AgentCF 将协同过滤（CF）推荐中的用户和物品同时建模为 LLM 驱动的自主智能体。用户 Agent 和物品 Agent 通过自主交互进行协同学习，Agent 通过"反思"机制——比较自身行为预测与真实历史记录的差异——来不断修正行为模式，从而弥合语言建模与行为建模之间的鸿沟。

**局限性**：AgentCF 将推荐视为一个纯粹的协同过滤问题，不涉及任何领域特定的上下文特征（用餐时段、营养需求、配送物流）。其 Agent 间的交互是"用户-物品"二元关系，缺乏对外部环境因素的建模能力。此外，AgentCF 运行在离线批处理模式下，不支持实时推荐。

**本项目的改进**：FoodMate-AI 在协同过滤基础上引入了**领域特定的多维度智能体分工**——环境感知智能体负责天气/交通/时段分析，画像智能体负责用户偏好与意图分析，协同过滤智能体负责 NCF 神经协同过滤，决策智能体负责多因素加权排序。这种"专业分工 + 协作融合"的架构远比 AgentCF 的"用户-物品"二元交互更适合外卖推荐这一高度情境化的场景。

### 2.1.3 DoorDash 多臂老虎机推荐平台

**来源**：DoorDash Engineering Blog, *Experimentation at DoorDash with a Multi-Armed Bandit Platform*, 2023.

**技术路线**：DoorDash（美国最大外卖平台之一）在其推荐系统中部署了基于 Thompson Sampling 的多臂老虎机平台，用于菜系推荐和搜索结果排序。系统使用 Beta 分布对每个推荐选项的转化率进行贝叶斯建模，并通过区域数据进行热启动（Warm-Starting），解决新用户冷启动问题。

**局限性**：DoorDash 的 MAB 平台主要用于 A/B 实验的替代方案（即在线实验优化），其上下文信号仅包含用户的历史行为和地理位置，不涉及天气、交通、健康数据等外部环境因素。此外，该平台不具备多智能体协作能力，所有决策逻辑集中在单一算法模块中。

**本项目的改进**：FoodMate-AI 在 DoorDash 的 Thompson Sampling 基础上实现了四种 MAB 策略（UCB1、Thompson Sampling、ε-Greedy、Contextual Bandit），支持运行时动态切换。更关键的是，FoodMate-AI 的默认策略 Contextual Bandit 将天气（±0.65）、交通（±0.12）、运动后状态（±0.35）、心率（±0.25）、血氧（±0.30）、压力（±0.25）、睡眠质量（±0.25）、环境光照（±0.15）和用户绝对意图（+0.40）等 **50+ 维上下文信号**融入每个臂的奖励估计中，实现了远超 DoorDash 的上下文感知能力。

## 2.2 端侧大模型部署与推理优化

### 2.2.1 MobileLLM：面向端侧场景的亚十亿参数语言模型

**来源**：Liu Z, Zhao C, Iandola F 等. *MobileLLM: Optimizing Sub-billion Parameter Language Models for On-Device Use Cases*. ICML 2024. (arXiv: 2402.14905)

**技术路线**：MobileLLM 是 Meta 提出的面向移动端部署的小规模语言模型系列，强调"深而窄"（Deep-and-Thin）的架构设计优于简单的宽度缩放。MobileLLM-125M 在手机端可达到约 50 tokens/s 的推理速度，MobileLLM-350M 在多项基准上接近大模型性能。该工作的核心贡献是证明了在亚十亿参数规模下，模型深度比宽度对最终性能的影响更大。

**局限性**：MobileLLM 是一个通用目的的基础语言模型，未针对任何特定领域（如餐饮/营养）进行优化。其设计聚焦于模型架构创新，未探讨模型量化（GPTQ/AWQ/GGUF）、NPU 硬件加速或端云协同的推理分流策略。

**本项目的改进**：FoodMate-AI 在端侧部署了经过领域微调的 Qwen-2.5 模型，采用 Q8 量化（GGUF 格式），模型体积压缩至 ~50MB，通过 llama.rn 框架运行在手机本地。与 MobileLLM 的通用模型不同，FoodMate-AI 的端侧 LLM 专门为外卖意图提取任务设计了 System Prompt 和多轮对话格式，输出严格的结构化 JSON。此外，FoodMate-AI 内置了 5 个版本的 GGML-Hexagon 加速库（覆盖骁龙 8 Gen 1 至最新一代芯片），利用高通 NPU 硬件加速推理，这是 MobileLLM 未涉及的工程优化。

### 2.2.2 Apple Intelligence 端侧基础语言模型

**来源**：Apple Machine Learning Research. *Apple Intelligence Foundation Language Models*. Technical Report, July 2024.

**技术路线**：Apple Intelligence 在 iPhone 15 Pro 上部署了一个约 30 亿参数的端侧基础语言模型，通过量化感知训练（QAT）将权重压缩至 2-bit，搭配 LoRA 适配器的混合 2-bit/4-bit 配置，平均每权重仅 3.7 bit。该系统在 iPhone 的 Neural Engine 上实现了约 0.6ms 的首 token 延迟和 30 tokens/s 的生成速度。Apple 的核心设计原则是"能在端侧处理的绝不上云"，将隐私保护作为架构设计的第一优先级。

**局限性**：Apple Intelligence 是一个通用目的的端侧 AI 系统，不涉及特定垂直领域的任务设计。其端侧与云端的分工是基于任务复杂度的静态路由，缺乏基于上下文动态决策的智能路由机制。此外，Apple Intelligence 是闭源系统，不可在非 Apple 设备上复现。

**本项目的改进**：FoodMate-AI 借鉴了 Apple Intelligence "隐私优先"的设计理念，将用户语音和健康数据的处理**严格限制在端侧**——Vosk 离线语音识别 + 端侧 LLM 意图提取的级联管线确保原始语音和自然语言文本不经过任何网络传输，仅将脱敏后的 5 个结构化约束字段发送至云端。与 Apple 的静态路由不同，FoodMate-AI 实现了**基于置信度的动态路由**：NutriVision 模块中，EfficientNet-B0 的分类置信度 ≥ 60% 时仅发送文本查询 LLM（快速路径），< 60% 时回退到全图像多模态推理（精确路径），实现了延迟与精度的动态平衡。

## 2.3 LLM 驱动的智能定价技术

### 2.3.1 LLP：基于大语言模型的电商商品定价

**来源**：Wang H 等. *LLP: LLM-based Product Pricing in E-commerce*. arXiv: 2510.09347, 2025. 已部署于阿里巴巴闲鱼平台。

**技术路线**：LLP 是首个基于大语言模型的商品定价生成框架，采用两步流水线：（1）相似商品检索——从历史成交数据中检索同类商品作为参考；（2）LLM 推理定价——将参考价格、商品描述和成色信息输入 LLM，推理生成最优售价。模型优化采用两阶段策略：先通过监督微调（SFT）学习定价模式，再通过群体相对策略优化（GRPO）进行强化学习微调。在闲鱼平台的 A/B 测试中，LLP 将定价采纳率从 40% 提升至 72%。

**局限性**：LLP 面向二手商品的**静态一次性定价**，不涉及基于时间序列的动态调价。其定价对象是单个商品的初始标价，不考虑供需波动、时段效应、天气影响等实时因素。此外，LLP 的定价决策完全由 AI 自主完成，缺乏"人机协同审批"机制。

**本项目的改进**：FoodMate-AI 的 AI 动态定价系统解决的是**周期性动态调价**问题——基于过去 7 天的销量和营收趋势，对现有菜品进行价格调整建议。与 LLP 的一次性定价不同，FoodMate-AI 的定价流水线是一个持续运行的闭环：RabbitMQ 实时采集销售数据 → 每周触发 LLM 分析 → 生成 MARKDOWN/SURGE/MAINTAIN 三类策略 → 商家可配置自动审批阈值或人工逐条审批。这种**"AI 出主意、人来拍板"的人机协同模式**，既发挥了 LLM 的数据分析能力，又保留了商家对自身经营的最终决策权，比 LLP 的全自动模式更适合对价格敏感的餐饮行业。

### 2.3.2 基于深度强化学习的外卖多模式配送动态定价

**来源**：*Dynamic Pricing for Multi-Modal Meal Delivery Using Deep Reinforcement Learning*. Future Transportation (MDPI), Vol. 5, Issue 3, Article 112, 2025.

**技术路线**：该研究针对拥有多种配送模式（步行、骑行、驾车）的外卖平台，设计了基于深度强化学习（DRL）的动态定价机制。系统将配送定价建模为马尔可夫决策过程（MDP），状态空间包含当前订单量、配送员可用数、天气等因素，动作空间为不同配送模式的价格组合，奖励函数综合考虑平台收益和用户满意度。

**局限性**：该方法使用传统的 DRL（如 DQN/PPO），缺乏 LLM 的自然语言推理能力，无法理解和利用菜品描述、用户评价等非结构化文本信息。其定价对象是配送费而非菜品价格，且模型训练需要大量历史交互数据，冷启动困难。

**本项目的改进**：FoodMate-AI 将 LLM 的**自然语言推理能力**引入定价决策——AI 引擎不仅分析结构化的销量/营收数据，还能在 Prompt 中融入菜品名称语义（理解"红烧牛肉面"和"酸菜肉丝面"的品类差异）和市场背景假设（滞销品降价引流 vs 畅销品提价增利），生成**可解释的中文分析理由**。这种基于 LLM 的方案不依赖大量历史交互数据，仅需 7 天销售统计即可给出合理建议，有效解决了传统 DRL 方法的冷启动问题。

## 2.4 多模态食物识别与营养分析

### 2.4.1 FoodLMM：基于大型多模态模型的通用食物助手

**来源**：Yin Y, Qi H, Zhu B 等. *FoodLMM: A Versatile Food Assistant using Large Multi-modal Model*. IEEE Transactions on Multimedia, 2025. (arXiv: 2312.14991)

**技术路线**：FoodLMM 是首个面向食物领域的统一多任务大型多模态模型，基于 LLaVA+SAM 架构扩展，通过任务特定的 head tokens 同时支持食物分类、食材识别、食谱生成、营养估算和食物分割五大任务。训练采用两阶段策略：第一阶段在公开食物基准数据集上进行多任务学习，第二阶段在自建的 FoodDialogues 和 FoodReasonSeg 数据集上微调。

**局限性**：FoodLMM 是一个学术研究模型，模型体积巨大（数十亿参数），无法在移动端部署或进行实时推理。其定位是一个通用的食物理解模型，不涉及与推荐系统、定价系统的集成，也不具备根据用户个人健康标签进行个性化分析的能力。

**本项目的改进**：FoodMate-AI 的 NutriVision 模块采用**端侧 CV + 云端多模态 LLM 的混合架构**，而非 FoodLMM 的单一大模型方案。端侧 EfficientNet-B0 用于快速食物分类（< 500ms），置信度 ≥ 60% 时仅发送菜品名称到云端 LLM 查询营养信息（文本模式，20 秒内响应），置信度 < 60% 时才发送完整图片到 Gemini 2.0 Flash 多模态 API 进行深度分析。这种**基于置信度的动态路由策略**在保持分析精度的同时大幅降低了延迟和 API 调用成本。更重要的是，NutriVision 的分析结果直接融入推荐系统的上下文——例如识别出用户拍摄的菜单中含有过敏原时，推荐系统会自动过滤该餐厅，实现了营养分析与推荐决策的**端到端闭环**。

### 2.4.2 NutritionVerse：多模态膳食摄入估计方法实证研究

**来源**：Tai C A, Keller M, Nair S 等. *NutritionVerse: Empirical Study of Various Dietary Intake Estimation Approaches*. Proceedings of the 8th International Workshop on Multimedia Assisted Dietary Management (MADiMa '23), co-located with ACM Multimedia 2023, Ottawa, Canada, October 2023.

**技术路线**：NutritionVerse 项目提供了两个基准数据集：NutritionVerse-Synth（84,984 张合成食物图像）和 NutritionVerse-Real（889 张真实图像，覆盖 251 道菜品），每张图像包含多模态标注（深度图、实例掩码、语义掩码、营养成分标签）。研究在此基准上对比了基于分割的预测网络和直接预测网络两类膳食摄入估计方法，并探索了合成数据到真实数据的迁移学习效果。

**局限性**：NutritionVerse 聚焦于数据集构建和算法基准评测，属于学术基础设施层面的工作，不涉及面向终端用户的产品级系统设计。其使用的图像来自受控的实验室环境，与真实外卖场景中用户在各种光线条件下拍摄的菜单/菜品照片存在显著的领域差距（Domain Gap）。

**本项目的改进**：FoodMate-AI 的 NutriVision 直接面向**真实外卖场景**——用户在餐厅现场拍摄的菜单照片或从相册选取的菜品图片，光线、角度、清晰度千差万别。系统通过 Gemini 2.0 Flash 的强泛化多模态能力和防御性响应标准化（`_standardize_response()`兼容 `items`/`dishes`/`menu_items` 三种命名、自动转换逗号分隔字符串为列表）来应对这种场景多样性。此外，NutriVision 的分析结果不是孤立的信息展示，而是与用户的健康标签（如"花生过敏""控糖"）实时交叉校验，生成**个性化的过敏原警告和 Top-3 推荐**。

## 2.5 隐私保护的端云协同推荐架构

### 2.5.1 SpFedRec：基于分裂联邦学习的隐私保护大规模推荐模型

**来源**：*A split-federated learning and edge-cloud based efficient and privacy-preserving large-scale item recommendation model*. Journal of Cloud Computing (Springer), 2023.

**技术路线**：SpFedRec 提出了一种结合分裂学习（Split Learning）和联邦学习（Federated Learning）的混合架构，用于在保护用户隐私的前提下训练大规模推荐模型。系统将深度推荐模型按层切分：底层（用户特征嵌入层）运行在边缘设备上，上层（交叉网络和输出层）运行在云端。边缘设备仅上传中间激活值（而非原始数据），云端汇聚多个设备的梯度进行联邦聚合，从而在保护原始数据隐私的同时完成全局模型训练。

**局限性**：SpFedRec 聚焦于**训练阶段的隐私保护**（联邦模型更新），但未解决**推理阶段的隐私问题**——用户在实际使用推荐服务时，仍需将查询请求（可能包含敏感的饮食偏好和健康信息）发送至云端。此外，该方案的通信开销较大（需要频繁传输中间激活值和梯度），在移动网络环境下的可行性存疑。

**本项目的改进**：FoodMate-AI 重点解决的是**推理阶段的实时隐私保护**，而非训练阶段。系统在手机端侧部署了完整的"语音识别 → 意图提取"管线（Vosk + Qwen Q8 GGUF），用户的语音音频和自然语言文本**全程不离开设备**。端侧处理完成后，系统仅将 5 个脱敏后的结构化约束字段（`forbidden_ingredients`、`required_temperature`、`preferred_tags`、`max_price`、`is_post_workout`）发送至云端推荐引擎。这种**"推理即脱敏"的端侧预处理架构**，在不增加通信开销的前提下（传输的 JSON 仅几百字节），实现了比联邦学习更直接、更彻底的隐私保护——敏感数据从始至终不存在于云端。

### 2.5.2 联邦学习综述：进展与开放问题

**来源**：Kairouz P, McMahan H B, Avent B 等. *Advances and Open Problems in Federated Learning*. Foundations and Trends in Machine Learning, 2021, 14(1-2): 1-210.（该综述由 Google 主导、58 位作者联合撰写，截至 2025 年被引超 8000 次，是联邦学习领域引用量最高的综述论文）

**技术路线**：该综述系统梳理了联邦学习的全生命周期，涵盖本地训练、安全聚合、全局模型更新、差分隐私、通信压缩等核心技术，并深入讨论了设备异构性（不同手机算力差异）、数据异构性（Non-IID 数据分布）和系统鲁棒性等在移动端落地的关键挑战。文中明确指出，联邦学习在保护训练数据隐私方面取得了显著进展，但对推理阶段的隐私保护讨论相对有限。

**局限性**：该综述聚焦于**训练阶段**的隐私保护范式（模型参数不出端），但未深入讨论**推理阶段**的隐私问题——即用户在实时使用 AI 服务时，查询请求本身可能包含的敏感信息（如饮食偏好、健康数据）如何保护。此外，综述未给出面向特定垂直领域（如外卖/餐饮推荐）的工程实践方案。

**本项目的改进**：FoodMate-AI 从工程实践角度回应了联邦学习综述提出的两大挑战：（1）**设备异构性**——通过内置 5 个版本的 GGML-Hexagon NPU 加速库（V69/V73/V75/V79/V81），自动检测设备芯片型号并加载对应的加速库，适配从骁龙 8 Gen 1 到最新一代的不同手机；（2）**隐私保护范式创新**——区别于联邦学习的"分布式训练"范式，FoodMate-AI 提出了**"端侧推理即脱敏"**的新范式：将隐私保护的边界从"训练数据不出端"提升到"推理数据也不出端"，仅传输推理结果的最小必要子集（脱敏约束 JSON），在架构层面从根本上消除了数据泄露风险。
