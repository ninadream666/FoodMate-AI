> 拉取最新代码后，**必须按以下步骤操作**，否则项目无法正常编译/运行。
> 本节涵盖了第一章至第十五章所有改动所需的操作。

### 第一步：前端依赖安装

```bash

cd frontend
npm install @react-native-community/netinfo --legacy-peer-deps

# 2. 清缓存重启 Metro（确保所有代码改动生效）
```

### 第二步：后端全量重建

```bash
# 必须全量重建！涉及：
# - docker-compose.dev.yml 健康检查全部从 curl 改为 wget/python（第十五章）
# - order-service: 商家接单API + merchantId兼容 + 佣金Feign路径修复 + 订单完成结算
```

### 常见问题排查

| 现象 | 原因 | 解决 |
|------|------|------|
| 服务状态 `(unhealthy)` | 旧代码用 curl 做健康检查，容器没有 curl | 确认拉了最新代码后 `docker compose up -d --build` |
| 推荐页"暂无推荐" | 推荐服务 ML 模型还在加载（约2分钟） | 等 `ps` 显示 healthy 后刷新 |

| 个人中心本月消费/订单数为0 | 旧代码未兼容 Page 对象和 status 对象格式 | 确认拉了最新前端代码 |

## 一、前端网络层优化

### 1.1 apiClient.js — 核心网络层升级

**文件**: `frontend/src/services/apiClient.js`

**改动内容**:

#### 分级超时
不同服务类型使用不同超时时间，避免图片上传被短超时截断：
```javascript
const SERVICE_TIMEOUT = {
    nutrivision: 120000,   // 图片分析：120s（含上传+AI推理）
    recommendation: 30000, // 推荐服务：30s（含外部API调用）
    default: 15000,        // 普通接口：15s
};
```

#### 指数退避重试
GET 请求失败后按 1s → 2s → 4s 退避重试；POST 等非幂等请求不重试（防止重复下单）：
```javascript
const RETRYABLE_STATUS_CODES = new Set([408, 429, 500, 502, 503, 504]);
const NON_RETRYABLE_METHODS = new Set(['POST', 'PATCH']);
// 退避公式
const backoffMs = Math.min(1000 * Math.pow(2, attempt - 1), 8000);
```

#### 请求取消 (AbortController)
支持按服务取消进行中的请求，页面卸载时自动释放带宽：
```javascript
export const cancelServiceRequests = (urlKey) => { ... };
export const cancelAllRequests = () => { ... };
```

#### 其他
- 请求头添加 `Accept-Encoding: gzip, deflate`，配合后端压缩
- 新增 429 状态码友好提示："请求过于频繁，请稍后再试"
- 响应拦截器打印响应体大小（KB），方便排查大 payload

## 二、首屏推荐编排架构（重要）

### 2.1 为什么之前会报错

推荐服务（8087 端口）是单个 Python 进程，之前前端同时发 3 个请求到它：

```
testAPIConnection() ──→ 8087 /agents/recommend   (完整AI编排，需要10-15秒)
fetchWeather()      ──→ 8087 /mcp/tool/weather    (天气查询，需要2-3秒)
loadData()          ──→ 8087 /agents/recommend    (完整AI编排，需要10-15秒)
                         │
                         ▼
              推荐服务排队处理，第2、3个请求等太久
                         │
                         ▼
              前端 15 秒超时 → apiClient.js 第99行报错 undefined
```

### 2.2 重新编排后的架构

**核心原则**: 天气、推荐、路况都走 8087 端口，**前端只发 1 次请求**，后端内部一站式编排。

#### 改动前（3 次并发打 8087，互相抢资源）

```
HomeScreen 初始化
  │
  ├─ [并行] loadUser()
  ├─ [并行] 获取定位
  ├─ [并行] testAPIConnection()   ← 打 8087（推荐服务）
  └─ [并行] 加载忌口
       │
       ▼ 定位完成后
  loadData(loc)
       │
       ├─ fetchWeather()           ← 又打 8087（天气接口）
       │    同时
       └─ 推荐请求                  ← 又打 8087（推荐接口）

  = 总共 3 次请求同时打 8087，推荐服务处理不过来 → 超时报错
```

#### 改动后（只打 1 次 8087，不再超时）

```
HomeScreen 初始化
  │
  ├─ [并行] loadUser()            ← 不打 8087
  ├─ [并行] 获取定位               ← 不打 8087（系统GPS）
  └─ [并行] 加载忌口               ← 打 8086（profile-service）
       │
       ▼ 全部就绪后
  loadData(loc)
       │
       └─ 1 次推荐请求 → 8087，携带：位置、健康数据(35字段)、忌口列表
            │
            │  后端收到后，ContextAgent 内部并行处理（asyncio.gather）：
            │  ├─ 天气查询 → 和风天气 QWeather API（真实温度、天气状况）
            │  ├─ 路况查询 → 高德地图 API（拥堵指数、拥堵等级）
            │  ├─ 日期/节日判断 → calendar_service + Python本地
            │  │   ├─ 是否周末：calendar API → 兜底 Python weekday()
            │  │   ├─ 是否节假日：calendar API（is_holiday）
            │  │   ├─ 节日名称：calendar API（holiday_name，如"清明节"）
            │  │   └─ 用餐时段：Python datetime.now().hour 判断
            │  │       6-10点=breakfast, 11-14=lunch, 14-17=afternoon_tea
            │  │       17-21=dinner, 其他=night_snack
            │  └─ 附近餐厅搜索 → 高德 POI API（20km半径，最多50家）
            │
            │  ProfilerAgent：
            │  └─ 用户画像 → profile-service（偏好、历史）
            │
            │  DecisionAgent：
            │  ├─ 忌口硬过滤：根据前端传入的 allergies 排除餐厅
            │  │   └─ "花生过敏"→排除含"花生/坚果"的餐厅
            │  │   └─ "不吃辣"→排除"川菜/火锅/湘菜/麻辣"
            │  │   └─ "痛风忌高嘌呤"→排除"海鲜/内脏/火锅"
            │  ├─ 健康数据加权：运动后→轻食加分，睡眠不足→咖啡加分
            │  ├─ MAB 排序：UCB1/Thompson 算法综合排序
            │  └─ AI 推荐语 → DeepSeek LLM（结合天气/健康/时段生成个性化理由）
            │
            ▼ 返回响应
       前端从响应的 context 字段提取天气数据 → 更新天气 UI
       前端展示推荐列表
```

### 2.3 改动前后对比

| 指标 | 改动前 | 改动后 |
|------|--------|--------|
| 前端请求 8087 次数 | 3 次并发 | **1 次** |
| 推荐服务排队 | 有（互相抢资源） | 无 |
| 报错概率 | 高（超时） | 基本消除 |
| `testAPIConnection()` | 多发一次推荐请求 | **删除** |
| `fetchWeather()` | 前端单独打 8087 天气接口 | **不再调用**，后端内部自己查 |
| 天气 UI 数据来源 | 前端 → 8087 天气接口 | 推荐响应 → context.weather |
| 天气参与推荐决策 | 前端查天气 → 传给后端（可能为 null） | 后端 ContextAgent 自己查，**100% 参与** |
| 首屏总耗时 | 定位(1-3s) + 天气排队 + 推荐(10-15s) | 定位(1-3s) + 推荐(10-15s) |
| 缓存命中时 | 同上 | **0 秒**（5分钟内同位置直接用缓存） |

### 2.4 推荐请求携带的完整上下文

| 上下文 | 来源 | 怎么传给后端 |
|--------|------|-------------|
| 位置（经纬度、地址） | 前端 GPS 定位 | `latitude`, `longitude`, `address` |
| 健康数据（35个字段） | 前端 OPPO 健康 SDK | `healthContext`（步数、心率、睡眠、血氧、压力等） |
| 忌口/过敏原 | 前端健康页面用户设置 | `allergies` 数组（如 ["花生过敏", "不吃辣"]） |
| 上次天气缓存 | 前端 state | `weatherContext`（辅助，后端也会自己查） |
| 天气（真实） | 后端 ContextAgent → 和风天气 API | 后端自己查，不需要前端传 |
| 路况 | 后端 ContextAgent → 高德地图 API | 后端自己查 |
| 日期/节日/时段 | 后端 ContextAgent → calendar_service | 后端自己查 |
| 附近餐厅 | 后端 RetrievalAgent → 高德 POI API | 后端根据经纬度自己搜 |
| 用户画像 | 后端 ProfilerAgent → profile-service | 后端根据 userId 自己查 |

## 八、商家接单功能（新增）

### 8.1 背景

此前系统中**缺少商家接单环节**。用户下单支付后订单状态为 `PAID`，但没有 API 将其推进到 `CONFIRMED`。这导致：
- 用户取消订单时，因订单始终为 `PENDING/PAID`，系统直接执行退款，不会进入 `CANCEL_PENDING` 状态
- 商家工作台的"退款审批"页面始终为空（它查询的是 `CANCEL_PENDING` 状态的订单）
- 商家无法感知新订单到达

### 8.2 订单状态流转（完整）

```
PENDING → (用户支付) → PAID → (商家接单) → CONFIRMED → (开始制作) → PREPARING
                         ↓                                              ↓
                    (商家拒单)                                     (备餐完成)
                         ↓                                              ↓
                    CANCELLED (全额退款)                              READY
                                                                        ↓
                                                                   (已出餐)
                                                                        ↓
                                                                   DELIVERED → COMPLETED

用户在 PAID 之后的状态取消订单 → CANCEL_PENDING → 商家审批 → CANCELLED/恢复原状态
```

### 8.3 后端改动

#### order-service

**`OrderService.java`** — 新增 4 个方法：

| 方法 | 功能 | 状态流转 |
|------|------|----------|
| `acceptOrder(orderId, merchantId)` | 商家接单 | PAID → CONFIRMED |
| `rejectOrder(orderId, merchantId, reason)` | 商家拒单（全额退款） | PAID → CANCELLED |
| `updateOrderProgress(orderId, merchantId, newStatus)` | 更新订单进度 | CONFIRMED→PREPARING→READY→DELIVERED→COMPLETED |
| `getMerchantOrdersByStatuses(merchantId, statuses)` | 查询商家指定状态的订单 | — |

内置状态流转校验 `isValidTransition()`，防止非法跳转（如 PAID 直接到 READY）。

## 九、商家订单查询 merchantId 不匹配修复 + profile-service 403 修复

### 9.1 商家订单查询查不到数据（核心 Bug）

#### 问题

商家在"订单管理"页面看不到任何待接单的订单，即使用户已经下单并支付。

#### 根因

系统中存在**两套 merchantId**：

| ID 类型 | 示例 | 来源 | 存在位置 |
|---------|------|------|----------|
| 数据库主键 | `1`, `2` | PostgreSQL 自增 | `merchants.id` |
| 外部 ID | `B0LDM1F2K5` | 高德地图 POI API | `merchants.external_id` |

用户下单时，前端传的 `merchantId` 是推荐服务返回的餐厅 ID（即**外部 ID**），所以订单表 `orders.merchant_id` 存的是 `B0LDM1F2K5`。

但商家工作台通过 `merchantService.getAllMyMerchants()` 获取的是**数据库主键** `1`，查询订单时传的是 `1`。

`WHERE merchant_id = '1'` 匹配不到 `merchant_id = 'B0LDM1F2K5'` 的订单 → 结果为空。

**退款审批**同样受此问题影响。

#### 修复方案

在查询和操作订单时，**同时使用数据库主键和外部 ID** 进行匹配。

#### 具体改动

**order-service:**

`OrderRepository.java` — 新增支持多 ID 查询的方法：
```java
List<Order> findByMerchantIdInAndStatusInOrderByCreatedAtDesc(List<String> merchantIds, List<OrderStatus> statuses);
List<Order> findByMerchantIdInAndStatus(List<String> merchantIds, OrderStatus status);
```

`OrderService.java` — 新增 2 个方法：
- `getMerchantOrdersByMultipleIds(List<String> merchantIds, List<OrderStatus> statuses)` — 支持多 ID 查询待处理订单
- `getPendingRefundOrdersByMultipleIds(List<String> merchantIds)` — 支持多 ID 查询待审批退款

`OrderInternalController.java` — `pending-orders` 和 `pending-refunds` 接口新增可选参数 `externalId`：
```
GET /orders/internal/merchant/{merchantId}/pending-orders?externalId=B0LDM1F2K5
GET /orders/internal/merchant/{merchantId}/pending-refunds?externalId=B0LDM1F2K5
```
收到后将两个 ID 合并为列表，调用多 ID 查询方法。

**merchant-service:**

`OrderServiceClient.java`（Feign 客户端）— `getPendingOrdersByMerchant` 和 `getPendingRefundsByMerchant` 新增 `@RequestParam externalId` 参数。

`MerchantOrderController.java` — 新增 `resolveOrderMerchantId(Merchant)` 辅助方法：
```java

private String resolveOrderMerchantId(Merchant merchant) {
```


### 9.2 profile-service 403 错误修复

#### 问题

每次进入餐厅详情页，控制台报 `403 http://127.0.0.1:8086/profile/history`。

#### 根因

profile-service 的 `JwtAuthenticationFilter` **缺少 try-catch**。当 JWT 解析抛出异常（过期、签名错误等）时，异常未被捕获，Spring Security 上下文中没有设置认证信息，`SecurityConfig` 的 `anyRequest().authenticated()` 拒绝请求返回 403。

#### 修复

`JwtAuthenticationFilter.java` — 用 try-catch 包裹 JWT 解析逻辑：
```java
try {
    final String jwt = authHeader.substring(7);
    final String username = jwtUtil.extractUsername(jwt);
    // ... 验证和设置认证
} catch (Exception e) {
    // Token 过期/无效时静默跳过，由 SecurityConfig 返回 401/403
}
filterChain.doFilter(request, response);
```

## 十一、退款审批 403 修复（merchantId 类型不匹配）

### 11.1 问题

商家在"退款审批"页面点击"同意退款"时，返回 403 "请求被拒绝"。

### 11.2 根因

`MerchantRefundService.approveCancellation()` 中验证订单归属的逻辑：

```java
// 从订单数据中取 merchantId，尝试转为 Long
Long orderMerchantId = parseLongSafe(orderData.get("merchantId"));
if (orderMerchantId == null || !orderMerchantId.equals(merchantId)) {
    return 403 "该订单不属于当前商家";
}
```

订单表中 `merchantId` 存的是外部 ID（如 `B0LDM1F2K5`），`parseLongSafe("B0LDM1F2K5")` 无法转为 Long → 返回 `null` → 第一个条件 `orderMerchantId == null` 为 true → **直接返回 403**。

这是第九章"merchantId 不匹配"问题在退款审批流程中的又一个表现，但位于不同的代码路径（`MerchantRefundService` 而非 Controller 层），所以之前的修复未覆盖到。

### 11.3 修复

**`MerchantRefundService.java`：**

方法签名新增 `externalId` 参数，商家归属校验改为字符串比较，同时匹配数据库主键和外部 ID：

```java
// 修复前：只能匹配数字 ID，外部 ID 直接 null → 403
Long orderMerchantId = parseLongSafe(orderData.get("merchantId"));
if (orderMerchantId == null || !orderMerchantId.equals(merchantId)) { → 403 }

// 修复后：字符串比较，同时匹配主键和外部 ID
String orderMerchantIdStr = orderData.get("merchantId").toString();
boolean isOwner = orderMerchantIdStr.equals(String.valueOf(merchantId))
        || (externalId != null && orderMerchantIdStr.equals(externalId));
if (!isOwner) { → 403 }
```

**`MerchantRefundController.java`：**

从 Merchant 实体获取 `externalId`，传给 `approveCancellation`：

```java
Merchant merchant = merchantOpt.get();
return refundService.approveCancellation(merchantId, orderId, approved, rejectReason, merchant.getExternalId());
```

```tsx
if (isFocused) {

    loadOrders(false);

}
```

## 十五、智能推荐排序优化 + 推荐语多样性 + 服务健康检查修复

### 15.1 不同天气/健康场景下推荐排序不变

#### 问题

无论模拟高温、大雨还是运动后，推荐的餐厅排序几乎不变。高温天应该优先推荐轻食/冰饮，大雨天应该推荐火锅/热汤，运动后应该推荐蛋白质/轻食——即使这些餐厅距离较远。

#### 根因

`ContextualBanditStrategy._calculate_contextual_score()` 中，距离权重固定为 0.25，与菜系匹配权重 0.20 相当。上下文加成（`context_bonus`）虽然存在（最大约 0.35），但不足以压过近距离餐厅的天然高分。结果是：**近距离的不匹配餐厅始终排在远距离的匹配餐厅前面**。

#### 修复

新增**动态权重机制**。检测到强上下文信号时自动调整各因素权重：

```python
# 强上下文信号：极端天气(≥30°C/≤10°C)、大雨、运动后、健康需关注、高压力、睡眠不足
has_strong_context = (
    temperature >= 30 or temperature <= 10
    or is_bad_weather
    or is_post_workout
    or overall_health == "需关注"
    or pressure >= 70
    or sleep < 6h
)
```

| 因素 | 默认权重 | 强上下文权重 | 变化 |
|------|---------|------------|------|
| 距离 | 0.25 | 0.08 | 大幅降低 |
| 评分 | 0.25 | 0.22 | 略降 |
| 价格 | 0.20 | 0.15 | 略降 |
| **菜系匹配** | **0.20** | **0.40** | **翻倍** |
| 配送时间 | 0.10 | 0.15 | 略升 |

同时，最终得分公式中的可变分乘数从 0.50 提升到 0.65（强上下文时），进一步拉大菜系匹配的分差。

### 15.2 推荐语单一
